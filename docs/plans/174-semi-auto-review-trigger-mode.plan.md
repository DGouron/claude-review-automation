## Architectural verdict (anti-overengineering)

Before designing the layers, three points are worth pinning down because they materially shrink the file count.

1. **The chokepoint is NOT `triggerReview.usecase.ts`.** Grep shows it is only called by `handleReviewRequestPush.usecase.ts` (the followup orchestrator). The three real Claude-invocation sites are direct calls to `enqueueReview` from `frameworks/queue/pQueueAdapter.ts` in:
   - `src/modules/platform-integration/interface-adapters/controllers/webhook/gitlab.controller.ts:336` (followup branch)
   - `src/modules/platform-integration/interface-adapters/controllers/webhook/gitlab.controller.ts:592` (initial review)
   - `src/modules/platform-integration/interface-adapters/controllers/webhook/github.controller.ts:287` (followup branch)
   - `src/modules/platform-integration/interface-adapters/controllers/webhook/github.controller.ts:535` (initial review)
   - `src/modules/tracking/interface-adapters/controllers/http/mrTrackingAdvanced.routes.ts:153` (manual followup re-trigger from dashboard)

   Each of these passes an inline `async (job, signal) => {...}` processor that owns the Claude invocation. We must intercept the moment **before** these call `enqueueReview`. The cleanest chokepoint that touches all five sites with minimal blast radius is a new use case `gateClaudeInvocation` that wraps `enqueueReview`: when `triggerMode === 'semi-auto'` it persists a pending request and skips `enqueueReview`; when `full-auto` it calls `enqueueReview` exactly as today.

2. **The Job entity does not exist as a separate domain artefact.** `JobStatus` is an interface inside `src/frameworks/queue/pQueueAdapter.ts:43` with status `'queued' | 'running' | 'completed' | 'failed'`. The pending state lives **outside** the in-memory queue (because it must survive process restart and because confirming a pending request must STILL go through `enqueueReview`). So we do not extend the existing `JobStatus` union; instead we introduce a new entity `pendingReviewRequest` that is the persisted snapshot of an unconsumed Claude invocation.

3. **Desktop notification already exists** — `src/dashboard/modules/desktopNotifications.js` has `shouldNotifyDesktop`/`getDesktopNotificationPayload`. We only need to add one new notification kind (`reviewPendingConfirmation`) and wire the existing flow on the new dashboard state slice. No new framework code.

These three observations let us hit V0 in ~13 files (within the spec's WARN budget).

**Processor resolution note**: the processor callback contains closures over `deps.reviewContextGateway`, `deps.threadFetchGateway`, etc. — these cannot be serialised. Two viable strategies, pick **A** for V0:

- **A (chosen)**: in each of the 5 call sites, *before* invoking `gateClaudeInvocation`, register a named processor builder on a small in-memory registry keyed by `triggerSource + platform + jobType`. `confirmPendingReview` looks it up via `resolveProcessor`. Trade-off: pending requests created before a restart are restored from disk but cannot replay if the process restarts BEFORE confirmation (we re-bind the registry from `routes.ts` on boot, so this works as long as the same code is running). The spec only requires pending to survive restart; it does not require the FULL processor to be persisted. Confirming a pending request after restart re-reads the persisted `ReviewJob` and rehydrates the processor from the registry — this is sufficient.
- B (deferred): persist a "kind" enum (`initial-review`/`followup`/`manual-followup`) per platform and let a switch in `confirmPendingReview` re-derive the processor inline. Cleaner long term but adds 3 more named usecase wrappers. Defer to a later refactor.

## WIRING

**Composition root** — `src/main/routes.ts`:

```
1. Instantiate FileSystemPendingReviewRequestGateway (no constructor args, same pattern as budget gateway).
2. Build the processor registry: new ProcessorRegistry(); register the three named builders
   (initial-review, followup, manual-followup) before any handler.
3. Create the use case instances:
   - listPendingReviews   = new ListPendingReviewsUseCase({ gateway })
   - confirmPendingReview = new ConfirmPendingReviewUseCase({ gateway, queuePort, enqueue: enqueueReview, resolveProcessor: registry.resolve })
   - dismissPendingReview = new DismissPendingReviewUseCase({ gateway, queuePort })
4. Register pendingReviewsRoutes with those 3 use cases + PendingReviewPresenter.
5. Inject a gateClaudeInvocation factory into each existing controller dependency bag, configured with
   triggerMode = config.triggerMode ?? 'full-auto', the gateway, the registry, and a WS broadcaster.
6. Add a new WS broadcast method broadcastPendingChanged() in src/main/websocket.ts that re-emits
   the current pending list (same shape as state broadcast at line 47).
```

**Webhook dependencies**:

- `GitLabWebhookDependencies` and `GitHubWebhookDependencies` interfaces (defined in their respective controller files) get a new field `gateClaudeInvocation: GateClaudeInvocationUseCase`. The existing inline `enqueueReview` calls (4 of them across the two controllers) become `await gateClaudeInvocation.execute({ job, triggerSource, processor })`.
- `mrTrackingAdvanced.routes.ts` similarly receives `gateClaudeInvocation` via its options and rewrites the single `enqueueReview` call.

## RISKS

| ID | Risk | Likelihood | Mitigation |
|----|------|-----------|------------|
| R1 | **Processor cannot be serialised** — pending requests survive restart but the closures around `deps.threadFetchGateway`, etc. don't. | High by design | Strategy A above: rehydrate processors from a code-side registry at boot. Document that adding a new processor kind requires registering it in `routes.ts`. Acceptance test covers the restart path (read pending file → list returns it). |
| R2 | **`enqueueReview` deduplication may race with confirm**. When `confirmPendingReview` calls `enqueueReview` and the same `jobId` is already active, current behaviour returns `false`. | Medium | The use case explicitly checks `queuePort.hasActiveJob(jobId)` before delegating and returns `already-running` with the French message from the spec (`Cette review est déjà en cours`). |
| R3 | **Config hot-reload not supported**. `loadConfig()` caches via `cachedConfig` (`configLoader.ts:231`). Toggling `triggerMode` requires a restart. | Low | Spec out-of-scope item: *"Hot-reload of the trigger mode setting (a service restart after config change is acceptable)"*. | R4 | **Dashboard JS coupling** to WS state shape — adding a new section means the inline script in `index.html` must subscribe to `pending-changed` events. | Medium | The websocket already broadcasts `state` messages on every job lifecycle change; we add ONE more message type and ONE more update path in the script. | R5 | **Existing controller tests will need updating** (the GitHub and GitLab webhook controller tests mock `enqueueReview` directly; they will now need to mock `gateClaudeInvocation`). | Medium | List the test files to update in step 13 of the order. Test mocks become *simpler* (one collaborator instead of three). |
| R6 | **Persistence migration**: none — `pendingReviewRequest.schema.ts` is brand new; existing on-disk files are untouched. | None | n/a |
| R7 | **SPEC-173 positioning**.
